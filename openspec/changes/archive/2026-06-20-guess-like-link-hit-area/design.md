## Context

归档详情页中的“猜你喜欢”列表当前使用了 `block` 级别链接样式，导致整行都进入 hover 和 click 命中范围。这个行为在视觉上比实际文字内容更宽，会让用户误以为整条列表项都属于内容本体。

该变更只影响博客归档页的一个局部交互，不涉及数据、路由或布局结构的重构。

## Goals / Non-Goals

**Goals:**

- 让“猜你喜欢”列表的 hover 和 click 命中范围仅覆盖文字长度。
- 保留原有跳转行为和样式语义。
- 尽量只改动归档页局部实现，避免扩散到通用组件。

**Non-Goals:**

- 不调整推荐列表的数据来源、排序或过滤逻辑。
- 不修改其他页面的链接样式规范。
- 不引入新的交互控件或额外的客户端逻辑。

## Decisions

1. 直接收窄该页 `<Link>` 的显示方式，而不是改造 `Card` 或抽象一个新组件。
   - 理由：问题只存在于单个页面的单个列表，局部调整风险最低，且不会影响其他列表场景。
   - 备选方案：将该交互抽象成通用“文字链接列表”组件。
   - 取舍：通用组件复用性更高，但会增加抽象成本，并把一个局部样式问题扩散到公共层。

2. 保持链接语义不变，仅移除导致整行生效的块级展现类。
   - 理由：`Link` 仍然是正确的语义载体，问题来自展示宽度，而不是导航机制。
   - 备选方案：用 `<span>` 包裹文字再套一层更细粒度的可点击区域。
   - 取舍：新增包裹层可以更精确地控制 hover，但会增加 DOM 层级，且没有必要。

3. 不引入脚本级事件控制。
   - 理由：这个需求本质上是样式和结构边界问题，CSS/markup 即可解决。
   - 备选方案：通过 JavaScript 动态计算文本宽度再设定点击区域。
   - 取舍：脚本方案复杂、脆弱，而且会制造不必要的客户端负担。

## Risks / Trade-offs

- [风险] 去掉 `block` 后，如果父级列表样式依赖块级宽度，可能造成间距或换行表现变化。 → [缓解] 仅在该列表项内保留必要的内联或 `inline-block` 样式，并在页面中验证视觉结果。
- [风险] 文本过长时点击区域缩小，鼠标命中容错下降。 → [缓解] 保持合理的行高和列表间距，确保可读性优先于整行命中。

## Migration Plan

1. 修改归档详情页的“猜你喜欢”列表链接样式，使其不再占满整行。
2. 在本地验证 hover 样式、点击跳转和换行显示。
3. 如出现视觉回退，回退到原实现并重新评估是否需要单独的文本链接包装。
