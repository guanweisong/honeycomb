## Context

后台多个列表都需要展示状态：友情链接、用户管理、页面、文章、分类和评论。当前状态主要依赖文本呈现，虽然可读，但在密集表格中不够醒目，也缺少统一的视觉分层。此次变更不涉及数据结构、接口字段或状态枚举的调整，只需要把已有状态值映射为更明确的视觉标签。

## Goals / Non-Goals

**Goals:**
- 让后台多个列表的状态列使用 `Badge` 组件展示。
- 统一状态与颜色的映射，保证各列表的视觉一致性。
- 支持启用、禁用、已删除、已发布、待审核、已封禁、垃圾评论等状态的颜色区分。

**Non-Goals:**
- 不修改后端状态枚举或接口返回值。
- 不新增新的状态类型。
- 不改变状态文案本身。

## Decisions

1. 使用 `shadcn/ui` 的 `Badge` 作为统一承载组件。
   - 原因：组件库已在项目中作为基础 UI 方案，复用它可以保持风格一致并避免新增依赖。
   - 备选方案：直接使用自定义 `span` + className。该方案灵活，但会把样式逻辑散落到多个列表中，后续维护成本更高。

2. 使用自定义语义色映射而不是默认变体。
   - 原因：需求明确要求不同状态对应绿色、红色、琥珀色和灰色，需要把状态到颜色的映射显式固化。
   - 备选方案：仅使用 `Badge` 默认变体。默认变体无法稳定表达业务状态差异，且容易在不同页面产生不一致。

3. 将状态映射抽象为可复用的展示逻辑。
   - 原因：多个后台列表都需要同类展示，抽象后可以减少重复代码，并确保后续状态调整时只改一处。
   - 备选方案：在各页面分别实现。虽然实现快，但容易出现颜色、文案或边界状态不一致。

4. 保持状态值透传，展示层只负责视觉包装。
   - 原因：当前需求只涉及展示，不应把业务状态改写为展示状态，避免影响筛选、排序或导出等依赖原始值的逻辑。
   - 备选方案：在展示层重新映射状态文案。该方案会增加不必要的转换层，也更容易引入语义漂移。

5. 评论状态采用多徽章组展示。
   - 原因：评论状态本身可以同时包含多个标签，多状态数组更适合用一组徽章表达。
   - 备选方案：拼接成单行文本。该方案实现更简单，但可读性和扫描效率都更差。

## Risks / Trade-offs

- [不同列表的状态枚举并不完全一致] → 通过统一的展示映射函数分别兼容各自状态值，必要时为未知状态提供兜底样式。
- [自定义颜色可能与主题对比度不完全一致] → 选用已有设计系统中可接受的绿色、红色、琥珀色、灰色组合，并在页面实际效果中检查可读性。
- [状态展示逻辑散落在各个表格列定义中] → 抽取共享映射层或共享状态 `Badge` 组件，避免重复实现。
- [评论状态为多值展示] → 使用徽章组而不是单一徽章，确保每个状态都能被明确呈现。

## Migration Plan

1. 先在共享位置实现状态到 `Badge` 颜色的映射逻辑。
2. 分别替换友情链接列表、用户管理列表、页面列表、文章列表、分类列表和评论列表的状态列渲染。
3. 核对绿色、红色、琥珀色和灰色在各列表中的实际展示。
4. 如果需要回滚，只需恢复原来的文本渲染方式，不影响数据和接口。

## Open Questions

- 友情链接和用户管理列表是否存在更多历史状态需要一起纳入颜色映射。
- 当前项目中 `shadcn/ui Badge` 的自定义颜色实现方式是否已经有现成约定，需要与现有样式方案保持完全一致。
- 评论列表是否需要把多状态徽章组进一步抽成独立组件。
