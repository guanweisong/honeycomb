## Context

admin 侧分类目录列表当前按页展示，适合数据量较大时控制首屏和查询成本，但在这个场景里，管理员更常需要一次性浏览全部分类以便核对结构、查缺补漏或快速定位分类项。分页会增加切换成本，也会打断对整体层级和分布的判断。

本次变更聚焦于 admin 场景的列表呈现，不改变分类本身的数据模型和业务含义。

## Goals / Non-Goals

**Goals:**

- 让 admin 分类目录列表默认全量展示。
- 保持列表排序、字段和交互语义不变。
- 在不引入额外产品复杂度的前提下，降低管理员查看成本。

**Non-Goals:**

- 不修改分类目录的数据结构。
- 不引入新的筛选、搜索或树形折叠能力。
- 不对其他 admin 列表统一取消分页。

## Decisions

1. **仅针对分类目录列表取消分页**

   这样可以把影响面控制在单一管理页面，避免误伤其他数据量更大的后台列表。

   备选方案是将 admin 所有列表统一改为全量展示，但这会显著放大性能与交互风险，不符合当前需求范围。

2. **继续沿用现有列表排序与字段**

   需求核心是“展示全部”，不是“重构列表信息架构”。保留现有字段和顺序能减少学习成本，也能降低实现和回归测试成本。

   备选方案是同步优化字段密度或增加分组展示，但这会把单点需求扩展成信息架构改造。

3. **由列表查询层支持一次性返回完整数据**

   取消分页后，前端不能再依赖页码驱动查询。实现上应让该页面的查询路径直接获取完整列表，避免“前端拿到第一页但界面假装全量”的不一致状态。

   备选方案是前端多次拉取分页并合并展示，但这会增加请求复杂度、加载抖动和状态管理成本。

4. **保留性能边界评估为实现约束**

   由于条目数量可能上升，全量渲染需要在实现阶段验证是否存在明显卡顿或过慢的问题。若出现瓶颈，再考虑按数据量阈值或渐进渲染优化。

   备选方案是预先引入虚拟列表或懒加载，但在缺乏明确性能问题前，这会提高复杂度而未必带来实际收益。

## Risks / Trade-offs

- [风险] 全量展示导致接口响应变慢或页面渲染压力上升 → [缓解] 在实现和验证阶段评估数据规模上限，必要时再引入性能优化。
- [风险] 管理员误以为所有后台列表都不再分页 → [缓解] 仅在分类目录列表页面修改行为，并保持其他页面不变。
- [风险] 未来分类数量继续增长后全量展示不再适合 → [缓解] 通过本次变更先满足当前运营效率需求，后续如规模变化再重新评估分页策略。
