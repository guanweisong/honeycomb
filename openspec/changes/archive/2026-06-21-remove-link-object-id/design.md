## Context

友情链接内容已经迁移到 `page` 模板体系，旧的 `LINK_OBJECT_ID` 只用于把某个自定义评论对象强行识别为友情链接页，并生成固定的 `/links` 路径。这个约定和当前页面路由模型不一致，也让设置接口暴露了已经过时的实现细节。

## Goals / Non-Goals

**Goals:**
- 移除 `LINK_OBJECT_ID` 的读取和传递。
- 删除友情链接页的评论链接特判。
- 保持评论、邮件通知和设置接口的其余行为不变。

**Non-Goals:**
- 不重新设计评论系统的数据模型。
- 不调整友情链接页面本身的页面模板逻辑。
- 不变更现有评论邮件内容结构，除了去掉过时的友情链接页特判。

## Decisions

- 直接删除 `process.env.LINK_OBJECT_ID` 的所有调用点，而不是保留一个空值兜底。
  - 原因：该变量已经没有业务来源，保留兜底只会继续传播废弃约定。
  - 备选方案：保留字段但置空。这个方案会让前后端协议继续携带无意义字段，不符合清理目标。
- 让评论链接解析只依赖现有的 `postId`、`pageId`、`customId`。
  - 原因：这些字段已经足够表达当前评论归属，不需要再引入一个外部环境标识。
  - 备选方案：继续把友情链接页映射到固定 `/links`。这与 page 化后的路由模型冲突。
- 在设置接口中移除 `customObjectId.link`。
  - 原因：设置接口不应再暴露与友情链接页绑定的环境变量。
  - 备选方案：保留字段但不使用。这样会让调用方继续依赖废弃字段。

## Risks / Trade-offs

- [邮件或前端仍在读取 `customObjectId.link`] → 通过测试和代码搜索同步清理调用方，避免遗留依赖。
- [自定义评论链接展示结果变化] → 只影响友情链接页这个旧特例，其他页面评论链接保持不变。
