## Context

`admin/tag` 这类页面属于典型的后台表格页，当前问题不是数据本身，而是当列表变长后，整页滚动会把筛选、统计和批量操作区一起推离视野。这个变更只调整页面结构和滚动边界，不引入新依赖，也不改数据模型。

## Goals / Non-Goals

**Goals:**
- 让表格主体在数据量大时独立滚动。
- 保持页面的关键控制区在长列表场景下更容易使用。
- 让同类后台表格页采用一致的滚动模式。

**Non-Goals:**
- 不重做表格组件本身的交互细节。
- 不修改接口、数据加载方式或分页逻辑。
- 不强制所有后台页面都采用同一种布局，只有表格主体型页面纳入这次调整。

## Decisions

1. 将滚动容器下沉到表格主体，而不是继续依赖页面根节点滚动。
   这样能把“浏览数据”的行为限制在最接近数据的区域内，避免控制区随着长列表不断上移。替代方案是保留整页滚动并做 sticky 表头，但这无法解决筛选区和批量操作区一起离开的核心问题。

2. 把表格页拆成明确的稳定区和滚动区。
   稳定区承载筛选、统计、批量操作等元素，滚动区只负责行数据展示。相比把所有内容放进同一个容器，这种分层更容易统一到各个后台表格页，也更方便后续复用。

3. 通过布局约束控制高度，而不是依赖页面自然撑开。
   这样可以在数据增长时保持外层页面稳定，避免不同屏幕尺寸下出现不可预测的整页滚动。替代方案是仅依赖 CSS 自动溢出，但在复杂页面里会放大各页实现差异。

## Risks / Trade-offs

- [Risk] 独立滚动区可能让用户感觉“多了一层滚动” -> 通过保持外层页面稳定、让滚动区边界明确来降低认知负担。
- [Risk] 某些页面的控制区较高，导致可视数据区变小 -> 需要在具体页面上控制控制区高度和信息密度。
- [Risk] 不同分辨率下可视区域差异较大 -> 通过响应式高度约束和最小可用高度减少极端情况。
