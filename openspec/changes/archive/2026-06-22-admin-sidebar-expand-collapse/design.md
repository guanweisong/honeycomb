## Context

当前后台侧边栏承担主导航职责，但在菜单项较多时会持续占用较宽的布局空间，影响内容区展示。这个变更只涉及后台界面的导航交互，不引入新的后端接口，也不改变现有权限或路由结构。

## Goals / Non-Goals

**Goals:**
- 允许用户在展开与收起两种侧边栏状态之间切换。
- 在不同页面中保持一致的交互体验。
- 记住用户的侧边栏偏好，减少重复操作。

**Non-Goals:**
- 不调整现有菜单结构、权限模型或路由组织方式。
- 不引入新的存储服务或后端持久化接口。
- 不重新设计后台整体视觉体系，只做局部交互增强。

## Decisions

1. 使用前端本地持久化保存侧边栏状态。
   - 原因：这是纯 UI 偏好，使用本地存储即可满足“记住状态”的需求，避免增加后端复杂度。
   - 备选方案：服务端保存用户偏好。该方案虽然跨设备一致，但需要新增接口和数据维护成本，不符合当前变更范围。

2. 以“收起后保留图标、展开后显示完整文本”为默认交互模型。
   - 原因：这样可以在压缩宽度的同时保留可辨识性，并降低误操作风险。
   - 备选方案：收起后完全隐藏侧边栏。该方案能获得更多空间，但会显著增加导航恢复成本，不利于后台高频跳转。

3. 侧边栏状态在后台范围内统一生效。
   - 原因：用户通常期望在不同后台页面之间保持相同布局偏好，避免页面间状态不一致。
   - 备选方案：每个页面单独记忆状态。该方案会造成行为碎片化，增加理解成本。

4. 交互以单一切换按钮为主。
   - 原因：控制点清晰，符合后台布局的常见使用习惯，也便于测试和后续维护。
   - 备选方案：通过悬停或手势自动展开/收起。该方案可发现性更差，也更容易引入误触。

5. 窄屏环境默认初始化为收起态。
   - 原因：窄屏下内容区更紧张，默认收起可以避免导航占据过多空间。
   - 备选方案：窄屏仍默认展开。该方案对首次进入更直观，但会明显压缩主内容区，不利于后台阅读和操作。

## Risks / Trade-offs

- [收起状态下可见信息变少] → 通过保留图标、悬停提示或折叠分组标题降低识别成本。
- [本地持久化在清理浏览器数据后失效] → 这是可接受的偏好丢失，若后续需要跨设备同步再升级为服务端存储。
- [不同页面布局实现不一致] → 通过复用统一的后台布局容器和状态来源，减少页面级分叉。
- [窄屏下展开态仍然压缩内容区] → 允许用户主动收起，必要时在小屏尺寸下优先呈现收起态默认值。
