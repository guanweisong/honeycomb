## 背景

当前 `page` 模型只有标题、内容和状态，没有保存渲染模板信息。前台另有一个专门的 `/links` 路由，用写死的逻辑展示友情链接列表，这绕开了正常的页面流程。

这次变更会同时影响后台页面表单、页面数据模型、前台页面详情路由，以及现有的 links 路由；同时还需要考虑已有数据的迁移或默认值策略。

## 目标 / 非目标

**目标：**
- 在页面新建/编辑流程中支持模板选择。
- 将模板选择持久化到页面记录中。
- 当选择友情链接模板时，前台页面详情需要按不同方式渲染。
- 当模板驱动页面可覆盖该内容后，移除前台特殊处理的 links 页面。

**非目标：**
- 重新设计友情链接管理界面本身。
- 改变友情链接记录或评论行为的语义。
- 引入除这两个模板之外的任意自定义模板。

## 决策

- 将模板字段存储在 `page` 上，而不是从标题、别名或路由推导。
  - 原因：模板选择属于编辑内容，不属于路由元数据；并且它必须在标题或路径变更后保持稳定。
  - 备选方案：通过特殊页面 ID 或别名推断模板。拒绝原因是它把展示逻辑和 URL 形态强绑定了。

- 第一版使用固定的小型枚举式模板集合。
  - 原因：当前只需要 `default` 和 `friendly-links`，有限集合可以让校验、表单和渲染更简单。
  - 备选方案：使用自由输入的模板键。拒绝原因是容易产生非法组合，也会增加 UI 和校验复杂度。

- 默认模板继续沿用现有页面正文渲染路径。
  - 原因：可以把回归风险降到最低，并且对现有页面内容保持兼容，不需要额外的内容迁移成本。

- 将友情链接列表放进页面渲染逻辑里，而不是保留单独写死的路由。
  - 原因：页面本身成为模板选择的唯一来源，只有需要时才展示友情链接列表。
  - 备选方案：保留 `/links` 作为重定向包装层。拒绝原因是它仍然保留了特殊路由，并且会重复行为。

## 风险 / 权衡

- [已有页面没有显式模板数据] -> 默认值设为 `default` 可以避免破坏现有内容，但为了统一性仍可能需要一次迁移或回填。
- [前台路由变化可能影响书签] -> 如果移除 `/links`，上线时需要明确替代入口或重定向策略。
- [模板专属渲染可能偏离主页面布局] -> 复用当前页面正文布局，只在特殊模板下插入友情链接区块。
- [结构和界面改动会影响多个层] -> 按校验、API、UI、路由清理分阶段实现并分别验证。
